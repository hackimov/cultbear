# Техническое задание (ТЗ) на разработку интернет-магазина «CultBear»

**Версия:** 1.0  
**Дата:** 21 апреля 2026  

## 1) Цель проекта

Разработать полноценный интернет-магазин в РФ по продаже футболок с патриотической символикой Российской Федерации (герб, флаг, исторические и тематические принты).  
Сайт должен быть современным, быстрым, адаптивным, с белой визуальной стилистикой и приоритетом чёрных футболок в ассортименте.

Проект должен закрывать полный цикл: каталог, карточки товаров, складской учет, корзина, checkout, оплата (Точка Банк), уведомления в Telegram, личный кабинет, административная панель, роли и права, инфраструктура в Docker.

---

## 2) Область реализации

### 2.1 Что входит в MVP

- Каталог товаров с тематиками, поиском и фильтрами.
- Карточка товара с вариантами (модель/рукав, размер, цвет) и медиа (фото/видео).
- Складской учет по вариациям товара.
- Синхронная корзина для гостя и авторизованного пользователя.
- Checkout с подсказками адреса через DaData.
- Автосоздание пользователя при оформлении заказа.
- Оплата через эквайринг Точка Банк.
- Уведомления о заказе в Telegram-группу.
- Личный кабинет пользователя (история заказов и статусы).
- Админ-панель на Filament с RBAC (роли/права).
- Хранение медиа в S3-совместимом Object Storage.

### 2.2 Что не входит (если не согласовано отдельно)

- Мобильные приложения (iOS/Android).
- Мультиязычность и мультивалютность.
- Интеграции с маркетплейсами (Ozon/WB и т.д.).
- Сложная CRM/ERP интеграция.

## 3) Технологический стек (обязательный)

- **Backend:** Laravel 11 (LTS/последняя стабильная на момент старта).
- **Админ-панель:** Filament 3.x.
- **БД:** PostgreSQL 16+.
- **Кэш/сессии/очереди:** Redis 7+.
- **Файловое хранилище:** S3-совместимое (Yandex Cloud / Selectel / AWS S3).
- **Frontend:** Blade + Alpine.js + Tailwind CSS 3 (допустим Livewire точечно).
- **Аутентификация:** Laravel Fortify/Jetstream (или эквивалент) + Spatie Permission.
- **Очереди:** Laravel queue worker (Horizon рекомендован).
- **Контейнеризация:** Docker Compose.
- **Web-сервер:** Nginx + PHP-FPM.

### 3.1 Docker Compose (минимальные сервисы)

- `app` (php-fpm)
- `nginx`
- `postgres`
- `redis`
- `worker` (очереди)
- `scheduler` (cron/schedule:run)

Подъем окружения одной командой: `docker compose up -d`.

## 4) Бизнес-логика и пользовательские сценарии

## 4.1 Основные пользовательские роли

- **Гость:** просмотр каталога, добавление в корзину, оформление заказа.
- **Пользователь:** всё как у гостя + личный кабинет, история заказов.
- **Админ/менеджер:** управление товарами, заказами, тематиками, пользователями (по правам).

### 4.2 Ключевые сценарии

1. Пользователь заходит на сайт, выбирает тематику из меню.
2. Видит раскладку товаров (редактируемую из админки).
3. Переходит в карточку, выбирает параметры, добавляет в корзину.
4. На checkout вводит ФИО, email, телефон, адрес (подсказки DaData).
5. Если email новый — создаётся аккаунт, генерируется пароль, отправляется письмо.
6. Создаётся заказ, пользователь автоматически авторизуется.
7. Переход к оплате Точка Банк.
8. После успешной оплаты: статус заказа меняется, остатки списываются, сообщение уходит в Telegram.
9. Пользователь видит заказ в личном кабинете и отслеживает статус.

## 5) Функциональные требования

## 5.1 Главная страница

- Визуальный стиль: светлый (белый фон), контрастные чёрные футболки.
- Мобильный и десктопный адаптив.
- Верхнее меню: Главная, Тематики, О нас, Доставка, Контакты, Корзина, Профиль.
- На мобильных: гамбургер-меню с анимацией открытия.
- Hero-блок:
  - анимация при скролле: проявление чёрной основы футболки;
  - поверх — анимированное наложение герба РФ/принта;
  - реализация через CSS/JS (Intersection Observer + GSAP или эквивалент).
- Блоки ниже hero: Популярные тематики, Хиты продаж, Новинки.
- Допускается использовать референсные изображения (по предоставленной ссылке) только при соблюдении прав на контент.

### 5.2 Каталог и тематики

- URL тематики: `/themes/{slug}`.
- В меню отображаются активные тематики из БД по `sort_order`.
- В каждой тематике:
  - редактируемый шаблон раскладки (2/3/4 колонки на десктопе);
  - управляемый порядок карточек (drag-and-drop в админке);
  - закрепление товара в топе;
  - баннер/описание тематики.
- Фильтры: размер, модель (тип рукава), цена, наличие.
- Поиск по названию и артикулу.
- Пагинация или infinite scroll (фиксируется на старте проекта).

### 5.3 Понятия «Артикул», «Тематика», «Раскладка»

- **Артикул (SKU):** уникальный идентификатор дизайна/товара.
- **Тематика:** раздел каталога (например: Герб РФ, История, Сила армии).
- **Раскладка:** конфигурация визуального показа набора артикулов внутри тематики (порядок, тип сетки, приоритетные карточки, баннеры).
- **Модель:** параметр вариации товара (короткий рукав/длинный рукав/поло и т.д.).

### 5.4 Карточка товара

- Галерея изображений (основное + дополнительные ракурсы).
- Видео товара (опционально).
- Выбор вариаций: модель, размер, цвет.
- Актуальные цена и остаток по выбранной вариации.
- Блоки: Описание, Характеристики, Доставка, Отзывы (отзывы можно как модуль 2-го этапа).
- Кнопка «Добавить в корзину» с мгновенной реакцией UI.

### 5.5 Корзина

- Единая логика для гостя и авторизованного пользователя.
- До авторизации корзина хранится в сессии (Redis), после входа/регистрации выполняется merge.
- Обновление количества, удаление позиций, пересчет сумм в реальном времени.
- Проверка наличия на складе при изменении количества и перед оплатой.

### 5.6 Checkout и создание пользователя

- Поля: ФИО, email, телефон, адрес доставки.
- Адресные подсказки DaData Suggestions API.
- При отправке:
  - создается заказ со статусом `new`;
  - если пользователь не существует — создается аккаунт;
  - генерируется временный пароль (минимум 10 символов), отправляется письмо с установкой/сбросом пароля;
  - пользователь автоматически авторизуется.
- Оплата запускается после успешной валидации заказа.

### 5.7 Оплата (Точка Банк)

- Создание платежа через API/платежную ссылку.
- Сохранение `payment_id`, `payment_status`, технических метаданных.
- Поддержка webhook от банка:
  - проверка подписи/секретного ключа;
  - защита от повторной обработки одного события (idempotency);
  - обновление статуса заказа.
- Статусы минимум: `new`, `awaiting_payment`, `paid`, `processing`, `shipped`, `delivered`, `cancelled`, `payment_failed`.

### 5.8 Telegram-уведомления

- Отправка в группу только после подтверждения оплаты.
- Формат сообщения: № заказа, дата, ФИО, телефон, адрес, состав заказа, сумма, способ оплаты.
- Отправка через очередь, логирование результатов отправки.

### 5.9 Личный кабинет

- Разделы: профиль, история заказов, карточка заказа.
- В заказе: состав, сумма, адрес, текущий статус, таймлайн изменений.
- Доступ только к своим заказам.

### 5.10 Административная панель (Filament)

- Роли:
  - `super_admin` — полный доступ;
  - `admin` — товары/тематики/заказы/пользователи (ограничиваемо);
  - `manager` — заказы и остатки.
- Возможность назначать роль пользователю из админки.
- CRUD-модули:
  - товары и вариации;
  - тематики и раскладки;
  - заказы;
  - пользователи и роли/права;
  - настройки сайта;
  - медиа (S3).
- В модуле "настройки сайта" предусмотреть редактирование реквизитов ИП/ЮР лица:
  - наименование организации/ФИО ИП;
  - ИНН/КПП/ОГРН(ОГРНИП);
  - юридический и почтовый адрес;
  - контактный email и телефон.
- Реквизиты ИП/ЮР лица должны автоматически подставляться:
  - в подвал сайта;
  - в политику конфиденциальности;
  - в пользовательское соглашение.
- Дополнительно: массовые операции, импорт CSV (товары/остатки), экспорт заказов.

## 6) Структура данных (минимальная модель БД)

### 6.1 Основные таблицы

- `users` — пользователи.
- `roles`, `permissions`, `model_has_roles`, `role_has_permissions` — Spatie Permission.
- `themes` — тематики.
- `theme_layouts` (или JSON в `themes`) — параметры раскладки.
- `products` — базовые карточки товаров.
- `product_variants` — вариации по модели/размеру/цвету.
- `product_media` (или Spatie Media Library) — медиа-объекты.
- `carts`, `cart_items` — корзины.
- `orders`, `order_items` — заказы.
- `payments` — транзакции/состояние платежа.
- `telegram_notifications` — лог уведомлений.
- `settings` — глобальные настройки сайта.

### 6.2 Ключевые поля (обязательно)

- `products.article` — уникальный артикул.
- `product_variants.sku_variant` — уникальный SKU вариации.
- `product_variants.stock_quantity` — остаток на складе.
- `orders.status` + `orders.payment_status`.
- `orders.delivery_address_json` + нормализованные поля адреса.
- `orders.user_id` nullable только для legacy-кейсов (в целевом сценарии должен быть связан с пользователем).

### 6.3 Ограничения и целостность

- Уникальность `article`, `slug`, `sku_variant`.
- FK между заказами/позициями/вариациями.
- Нельзя оплатить заказ дважды (idempotency по webhook/event_id).
- Нельзя продать товар с остатком < 0.

## 7) Нефункциональные требования

- **Производительность:** целевое TTFB и кэширование каталога, lazy load медиа.
- **Адаптивность:** mobile-first, корректно на 360px+ и популярных desktop-разрешениях.
- **Безопасность:** CSRF, XSS protection, rate-limit на auth/checkout/webhook.
- **Надежность:** очереди для внешних интеграций, retry/persistent jobs.
- **Логирование:** централизованные логи ошибок и webhook-событий.
- **SEO:** ЧПУ, meta title/description, sitemap.xml, robots.txt, OpenGraph.
- **Доступность:** базовая семантика, контрастность, alt для изображений.

## 8) Хранение медиа в S3

- Все изображения/видео товаров хранятся вне контейнера в S3-совместимом хранилище.
- В БД сохраняются только ключи/метаданные объектов.
- Обязательны:
  - генерация превью/миниатюр;
  - WebP-конвертация для изображений;
  - ограничение размера и формата при загрузке;
  - lifecycle-политики для неиспользуемых файлов.
- Доступ к объектам:
  - публичные URL только для каталога (или signed URL при приватной политике);
  - закрытые бакеты для внутреннего контента.

## 9) Интеграции

### 9.1 DaData

- Использовать Suggestions API.
- Подсказки адреса обязательны в checkout.
- Обработка ошибок API и fallback на ручной ввод.

### 9.2 Точка Банк

- Интеграция через официальный API/SDK.
- Webhook endpoint с верификацией подписи.
- Логирование payload и статуса обработки.

### 9.3 Telegram Bot

- Отправка уведомлений в группу по `chat_id`.
- Шаблон сообщения хранить в конфиге/настройках.
- Очередь + retry при временных ошибках.

### 9.4 Email-уведомления

- Письмо о создании аккаунта/установке пароля.
- Письмо-подтверждение заказа.
- Письмо об изменении статуса заказа.

## 10) Права доступа (RBAC)

- Разграничение публичной части, пользовательской зоны и админ-панели.
- Только `super_admin/admin` могут назначать роли.
- Менеджер не может менять роли и системные настройки.
- Все действия в админке логируются (кто и что изменил, когда).
